# ClawdMate 部署与发布文档（Windows 桌面版）

## 目标
- 让普通用户不需要拉代码、不需要安装 Node/Git，也能直接安装使用 ClawdMate。
- 发布形态：Tauri 桌面安装包（`.exe` / `.msi`）。

## 推荐架构
- 客户端：Tauri 打包后的桌面应用。
- 数据与登录：线上 Supabase（生产环境）。
- 生图：走你自己的线上 API（不要把密钥硬编码在客户端）。

---

## 一、发布前必须完成的改造

### 1. 去掉开发地址硬编码
当前项目里出现了 `127.0.0.1:5173` 的开发地址逻辑。发布前必须改为“生产可用”模式：

- `src-tauri/src/lib.rs`
  - `dashboard` 窗口不要固定使用 `WebviewUrl::External("http://127.0.0.1:5173")`。
  - 生产模式应加载打包后的前端资源（应用内页面）。
- 前端里所有 `localhost` 跳转
  - 例如“回到主页”按钮，不应依赖本机 dev server。
  - 改为应用内路由跳转（如 `/app/dashboard`）或通过桥接唤起应用内 dashboard。

### 2. 区分开发环境与生产环境
- 开发：允许本地地址（Vite dev server）。
- 生产：只使用打包资源 + 线上服务地址。
- 建议使用环境变量控制：
  - `VITE_APP_ENV=development|production`
  - `VITE_SUPABASE_URL`
  - `VITE_SUPABASE_ANON_KEY`
  - 生图服务地址（如 `VITE_IMAGE_GATEWAY_URL`）

### 3. 生图 API 安全策略
- 不要把真实 API Key 放前端或 Tauri 可逆代码里。
- 应使用服务端中转 API：
  - 客户端只传 prompt / 数据。
  - 服务端持有真实密钥并调用上游模型。
  - 服务端做额度与限流（例如每用户每天最多 2 次）。

## 二、打包发布流程

### 1. 本地检查
在项目根目录执行：

```powershell
npm run build
```

确认前端构建通过。

### 2. 构建桌面安装包
```powershell
npm run tauri build
```

产物目录通常在：

- `src-tauri/target/release/bundle/`

你会拿到 `.exe` / `.msi` 安装包（取决于配置）。

### 3. 安装包分发
- 上传到官网下载页 / 网盘 / 企业文档。
- 用户下载安装后即可直接运行，不需要源码。

---

## 三、线上环境配置（生产）

### 1. Supabase（生产）
- 创建独立生产项目（不要和开发库混用）。
- 配置生产 URL 和 anon key。
- 检查 RLS 策略、表结构、索引是否完整。

### 2. 生图服务（生产）
- 部署你自己的图片生成中转服务。
- 必须包含：
  - 鉴权（用户身份）
  - 频控（每天 2 次等）
  - 错误信息归一化（前端可读）

### 3. 环境变量
- 在打包前确认生产变量已生效。
- 严禁把私钥提交到仓库。

---

## 四、发布验收清单

- [ ] 安装包能在“无开发环境”的 Windows 机器上安装并启动。
- [ ] 启动后不依赖 `localhost:5173`。
- [ ] 登录可用，Supabase 读写正常。
- [ ] 今日任务、历史记录、日报页面正常。
- [ ] 生图请求可成功，失败提示友好。
- [ ] 游客点击生成日报会触发登录与提示。
- [ ] 每用户每日生成次数限制生效（2 次）。

---

## 五、后续建议（上线后）

- 接入自动更新（Tauri Updater），用户无需重复手动下载安装。
- 接入错误监控（如 Sentry）。
- 维护一个 `CHANGELOG`，每次发布记录修复内容。
